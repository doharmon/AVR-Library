CC3000 HCI DATA messages

This page list the CC3000 modules HCI Data messages and explains their protocol fields. For basic understanding of the CC3000 protocol, please see Host Controller Interface page. The message field bytes in this page written in color lime denote example values for the specific message and thus they can have alternate values according to message content.

The content of this page applies to CC3000 Service Pack v1.12 (CC3000 firmware version 1.26).

Please note that this page currently does not display a full list, but only a selection of all possible CC3000 HCI Data messages.

HCI Data messages are used for transporting data to be sent or received over the wireless link or to read or write the CC3000 non-volatile memory (EEPROM). HCI Data messages can carry two types of payload information: arguments and data.


HCI_DATA_SEND (0x81)

HCI_DATA_SEND message is used to transmit a TCP packet to a socket. Before issuing a HCI_DATA_SEND message to CC3000, a socket should first be created using HCI_CMND_SOCKET (0x1001). Please note that a socket will be closed by CC3000 after 60 seconds of inactivity, thus a time between creating a socket and issuing HCI_DATA_SEND should not exceed this period! Use the Socket Handle returned by HCI_EVNT_SOCKET (0x1001) in the appropriate HCI_DATA_SEND argument field.

HCI_DATA_RECV (0x85)

Spiheader+ 02 85 18 1E00 00000000 0C000000 0600 0000 0000 0000 0000 0000 0000 0000 3132 3334 3500
example server sending 5 bytes "12345" with a tcp padding byte added as the 0600 show the length, though it only was 5 real data bytes. minimum tcp data block is 6, so if tcp timeouts it will send padding bytes.
If your 0x1004 cmd requested less than 6bytes you will only get those number of bytes but the padding bytes may show up next time.
So your parser have to take in to account padding bytes (zeros) as you will not be able to tell if it was a real zero or a padding zero.
Best, make your server transfers protocol (header+data etc) you come up with is at minimum 6bytes.

HCI_DATA_SENDTO (0x83)

HCI_DATA_SENDTO message is used to transmit a UDP packet to a socket. Before issuing a HCI_DATA_SENDTO message to CC3000, a socket should first be created using HCI_CMND_SOCKET (0x1001). Please note that a socket will be closed by CC3000 after 60 seconds of inactivity, thus a time between creating a socket and issuing HCI_DATA_SENDTO should not exceed this period! Use the Socket Handle returned by HCI_EVNT_SOCKET (0x1001) in the appropriate HCI_DATA_SENDTO argument field.

HCI_DATA_NVMEM_WRITE (0x90)

HCI_DATA_NVMEM_WRITE message is used to write to CC3000 NVMEM/EEPROM. HCI_DATA_NVMEM_WRITE can be used to write NVMEM/EEPROM areas for MAC-address, ServicePacks or user files (file ID's 4-6,10,12-15). The file ID entry to be written does need to be allocated first - see HCI_CMND_NVMEM_CREATE_ENTRY (0x0203) how to achieve this.

Note that the HCI Data message header does not explicitly include the length of data field bytes, instead CC3000 calculates this by subtracting the arguments length from the HCI payload length. The HCI_DATA_NVMEM_WRITE message has always exactly 16 argument bytes and in the example above 16 (this value is variable) data bytes are being written to NVMEM/EEPROM, resulting in a total HCI payload of 32 bytes.


When the CC3000 has processed the HCI_DATA_NVMEM_WRITE, it answers with an HCI_EVNT_NVMEM_WRITE (0x0202) event.


HCI_DATA_NVMEM_READ (0x91)

HCI_DATA_NVMEM_READ message is triggered by sending HCI_CMND_NVMEM_READ (0x0201). The direction of HCI_DATA_NVMEM_READ is always from CC3000 towards SPI master.

In the example message above, the MAC address from EEPROM is being replied, with the value in the data field (08:00:28:01:02:03). The six 32-bit arguments are not evaluated during HCI_DATA_NVMEM_READ by the TI host driver software from SDK.

The Organizationally Unique Identifier (OUI) of CC3000 MAC address (the first 3 bytes of MAC address) is always 08:00:28, as this OUI identifies Texas Instruments.

Example reading out the SmartConfig configuration vector

When CC3000 receives AES-128 encrypted SmartConfig WLAN association data from it's peer, it saves the information as 67 byte long vector in the NVMEM file ID 13 (NVMEM_SHARED_MEM_FILEID). It is up to user code to use this information, either to save to a profile using HCI_CMND_WLAN_IOCTL_ADD_PROFILE (0x0005) or just connect once to an access point using HCI_CMND_WLAN_CONNECT (0x0001).


The 67 byte long SmartConfig WLAN association vector contains:

  • 1 byte, SSID length in binary
  • 1 to 32 bytes, SSID name in ASCII
  • 1 byte, key field length in binary (always 32 (hex 0x20) irrespective of the real key/passphrase length)
  • 1 byte, WLAN security type (0=no security, 1=WEP, 2=WPA, 3=WPA2)
  • 32 bytes, key/passphrase, optionally encrypted using AES-128 with 16-byte seed key specified in peer SmartConfig program/app
  • Possibly some unused bytes at the end of 67 byte vector, if SSID length was less than 32.

The 32 byte long key/passphrase field contains in its first byte the key length if the key is less than 32 bytes long, otherwise the key/passphrase field is fully used to withhold the key.
