This page list the CC3000 modules HCI Command messages and explains their protocol fields. For basic understanding of the CC3000 protocol, please see Host Controller Interface page. The message field bytes in this page written in color lime denote example values for the specific message and thus they can have alternate values according to message content.

The content of this page applies to CC3000 Service Pack v1.12 (CC3000 firmware version 1.26).

Please note that this page currently does not display a full list, but only a selection of all possible CC3000 HCI Command messages.

HCI command messages are sent by SPI master towards CC3000 to trigger an action or data request, thus only MOSI direction HCI command messages are defined. Typically, CC3000 answers to command messages with a command complete event, whos event operation code is the same as the command operation code that triggered it.


HCI_CMND_WLAN_CONNECT (0x0001)

This command is used to instruct the CC3000 to associate with an access point.

Note that SSID and security key can both be maximally 32 bytes long. Altough the HCI_CMND_WLAN_CONNECT message includes BSSID field, connecting to BSSID is currently not supported (see CC3000 Errata).


Following WLAN security types are defined:

WLAN security Type code
Unsecured 0x00000000
WEP 0x00000001
WPA 0x00000002
WPA2 0x00000003

When using WEP-security, only ASCII key type is supported by CC3000. Thus the length of the key, when using WEP-security, can be either 5 or 13 characters (bytes). For more details about this, please see Wiki page about WEP-security.

When the CC3000 has processed the HCI_CMND_WLAN_CONNECT command, it answers with an HCI_EVNT_WLAN_CONNECT (0x0001) event. Note that it might take many seconds (even up to 10 seconds) for CC3000 to connect to an access point. Typically HCI_EVNT_WLAN_CONNECT (0x0001) event is followed by further event:

  • HCI_EVNT_WLAN_UNSOL_CONNECT (0x8001),

In case the CC3000 was configured in DHCP mode (see HCI_CMND_NETAPP_DHCP (0x2001) for more info about static or dynamic addressing), and if reply from DHCP server was received, CC3000 also emits:

  • HCI_EVNT_WLAN_UNSOL_DHCP (0x8010) event, which includes 5*4 (20) bytes of DHCP data (CC3000 client IPv4 address, Subnet, Gateway, DHCP and DNS IPv4 addresses) in the message arguments.

Note that all parameters, except the Interval[ms]/Disable, are saved into the CC3000 NVMEM/EEPROM. Setting the Interval[ms]/Disable parameter to 0, disables the scan. A value smaller than 1 second (1000) is not allowed.

The bitmask Channel Mask tells CC3000 which WLAN channels on the 2.4GHz band to scan. CC3000 allows to choose between channels 1 to 13. Note that in North America only channels 1-11 are allowed (channel mask value 0x000007FF), whereas most of the rest of the world all CC3000 supported channels (1-13) can be used (channel mask value 0x00001FFF). The TXpower field is not evaluated by CC3000 (at least not up to Service Pack 1.12 (firmware version 1.26)).


When the CC3000 has processed the HCI_CMND_WLAN_IOCTL_SET_SCANPARAM command, it answers with an HCI_EVNT_WLAN_IOCTL_SET_SCANPARAM (0x0003) event. The results of the scan can be read out using HCI_CMND_WLAN_IOCTL_GET_SCAN_RESULTS (0x0007) command. Please wait minimally for the duration of the longest set Channel Scan timeout, before calling the HCI_CMND_WLAN_IOCTL_GET_SCAN_RESULTS (0x0007) command, otherwise the query might reply with empty results.


HCI_CMND_WLAN_IOCTL_SET_CONNECTION_POLICY (0x0004)

This command is used to control if and how the CC3000 will try to automatically associate to an access point.


HCI_CMND_WLAN_IOCTL_SET_CONNECTION_POLICY has three user selectable parameters:

should_connect_to_open_ap if set, the CC3000 will try to connect to any access point it can find and associate with
should_use_fast_connect if set, the CC3000 will try to reconnect to last connected access point (works only, if last connection was successful)

HCI_CMND_WLAN_IOCTL_GET_SCAN_RESULTS (0x0007)

HCI_CMND_WLAN_SET_EVENT_MASK (0x0008)

Using this command the CC3000 can be instructed to omit transmitting some events, in case their reception is not desired. The transmission of all events are enabled by the default state of CC3000 at power-up/reset.

HCI_CMND_WLAN_SET_EVENT_MASK command can be used to disable following events:

Event name Event operation code Event description
HCI_EVNT_WLAN_UNSOL_CONNECT 0x8001 unsolicited connect
HCI_EVNT_WLAN_UNSOL_DISCONNECT 0x8002 unsolicited disconnect
HCI_EVNT_WLAN_UNSOL_INIT 0x8004 initialization done
HCI_EVNT_WLAN_UNSOL_TX_COMPLETE 0x8008 tranmission complete (virtual event; to be implemented in the host driver)
HCI_EVNT_WLAN_UNSOL_DHCP 0x8010 DHCP report
HCI_EVNT_WLAN_UNSOL_PING_REPORT 0x8040 ping report
HCI_EVNT_WLAN_UNSOL_SMART_CONFIG_DONE 0x8080 smart config done
HCI_EVNT_WLAN_UNSOL_KEEPALIVE 0x8200 keepalive


Note that the HCI_EVNT_WLAN_UNSOL_TX_COMPLETE is an exception in the list above, as this event is not a real event originating from the CC3000. Instead the operation code 0x8008 is reserved for the transmission complete logic to be optionally implemented in the host driver code, in case this feature is desired.


4-byte long bit mask field is used to command CC3000 to disable transmitting one or more of the events in the above table. Multiple events are concatenated by logically OR-ing their operation code values. As an example, the Basic WiFi Application from the SDK disables the transmission of 3 events:

  • 0x8004 HCI_EVNT_WLAN_UNSOL_INIT
  • 0x8040 HCI_EVNT_WLAN_UNSOL_PING_REPORT
  • 0x8200 HCI_EVNT_WLAN_UNSOL_KEEPALIVE

Logically OR-ing the 3 values 0x8004 || 0x8040 || 0x8200 results to 0x8244. Although the mask values are 2-byte long, the HCI_CMND_WLAN_SET_EVENT_MASK reserves for the mask 4-byte field (32-bits).

HCI_CMND_NETAPP_SET_DEBUG_LEVEL (0x2008)

This command is used to configure which messages CC3000 outputs over its UART debug channel. The user can select the messages to be sent out over the UART debug channel using a bitmask (with bit value 1 enabling the transmission):

Bit Message description Default state
0 Critical Enabled
1 Information Enabled
2 Core Enabled
3 HCI Enabled
4 Network stack Enabled
5 WLAN Enabled
6 WLAN driver Disabled
7 EEPROM Disabled
8 General Enabled

The CC3000 default (value 0x0000013F) transmits all data except WLAN driver messages and EEPROM messages. The value is not saved and thus will be restored to default after restarting CC3000.

HCI_CMND_SIMPLE_LINK_START (0x4000)

After power-up, the CC3000 module is initialized by sending 2 commands:

  • HCI_CMND_SIMPLE_LINK_START (0x4000), which initializes the CC3000 module internals, and
  • HCI_CMND_READ_BUFFER_SIZE (0x400B), which is used to correctly initialize the MCU host driver according to available buffers in CC3000.


The HCI_CMND_SIMPLE_LINK_START command parameter Patch Source tells CC3000 if and from where to load the firmware patches:

Patch source Code
CC3000 NVMEM/EEPROM (normal operation) 0x00
Force using patches from the host (the CC3000 does not load the patches from NVMEM/EEPROM) 0x01
No patches (the CC3000 does not load the patches from NVMEM/EEPROM) 0x02

The example Basic Wi-Fi Application only calls HCI_CMND_READ_BUFFER_SIZE once, during CC3000 initialization, and then keeps track of the number of free buffers in the code without repolling the value from CC3000.
