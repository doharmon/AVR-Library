This page list the CC3000 modules HCI Event messages and explains their protocol fields. For basic understanding of the CC3000 protocol, please see Host Controller Interface page. The message field bytes in this page written in color lime denote example values for the specific message and thus they can have alternate values according to message content.

The content of this page applies to CC3000 Service Pack v1.12 (CC3000 firmware version 1.26).

Please note that this page currently does not display a full list, but only a selection of all possible CC3000 HCI Event messages.

HCI Event messages are sent by CC3000 towards SPI master. They are either to acknowledge a command or to notify SPI master of asynchronous unsolicited event. Note that the event messages need to be clocked in by SPI master using READ protocol.

The transmission of some event messages can be disabled using HCI_CMND_WLAN_SET_EVENT_MASK (0x0008) command. Following events can be disabled:

  • HCI_EVNT_WLAN_UNSOL_CONNECT (0x8001), unsolicited connect event
  • HCI_EVNT_WLAN_UNSOL_DISCONNECT (0x8002), unsolicited disconnect event
  • HCI_EVNT_WLAN_UNSOL_INIT (0x8004), initialization done event
  • HCI_EVNT_WLAN_UNSOL_TX_COMPLETE (0x8008), transmission complete event (virtual event, not originating from CC3000, but from host driver code. See HCI_CMND_WLAN_SET_EVENT_MASK (0x0008) for more details on this).
  • HCI_EVNT_WLAN_UNSOL_DHCP (0x8010), DHCP report event
  • HCI_EVNT_WLAN_UNSOL_PING_REPORT (0x8040), ping report event
  • HCI_EVNT_WLAN_UNSOL_SMART_CONFIG_DONE (0x8080), smart config done event
  • HCI_EVNT_WLAN_UNSOL_KEEPALIVE (0x8200), keepalive event


HCI_EVNT_WLAN_CONNECT (0x0001)

HCI event operation code of HCI_EVNT_WLAN_CONNECT is the same as the calling command HCI_CMND_WLAN_CONNECT (0x0001). Note that it might take CC3000 many seconds for CC3000 to connect to an access point and answer with an HCI_EVNT_WLAN_CONNECT event (experimentation revealed time periods of up to 10 seconds).

The HCI_EVNT_WLAN_CONNECT SPI message is as follows:

The Result field is 0 for success, and the value will be negative (highest bit of 32-bit value set) in case of error. Please note that the result Success=0 only means that the connection profile has been successfully saved and NOT that CC3000 has associated with an access point.

The association with an access point is notified using two further events:

  • HCI_EVNT_WLAN_UNSOL_CONNECT (0x8001), and
  • HCI_EVNT_WLAN_UNSOL_DHCP (0x8010).

HCI_EVNT_WLAN_UNSOL_DHCP (0x8010) event includes in it's payload 5*4 (20) bytes of networking data (CC3000 client IPv4 address, Subnet, Gateway, DHCP and DNS IPv4 addresses). This event is sent after association with an access point, even if the CC3000 is configured to use static addressing instead of DHCP (see HCI_CMND_NETAPP_DHCP (0x2001) for more info on how to configure dynamic or static addressing).

The Profile slot field indicates the profile memory location, to which the profile was saved. CC3000 numbers the found networks with Network ID's. A HCI_EVNT_WLAN_IOCTL_GET_SCAN_RESULTS event outputs a single scan record at a time, starting with the highest Network ID. The user should keep calling HCI_CMND_WLAN_IOCTL_GET_SCAN_RESULTS (0x0007) command in order to read out also other found networks. When the Network ID field reaches 0, the Network ID's roll over and start again from the highest Network ID, which at the same time indicates the number of networks found.

Please note that the Network ID=0 will not contain scan record data, but will have all fields except Scan status at zero values (including the Result valid field, where 0=unvalid).

The Scan status field can have following values:

Value Hex value Description
0 0x00000000 Results are aged
1 0x00000001 Scan is valid
2 0x00000002 No results


Scan status 0 = Aged results will be output once if the scan results are queried using HCI_CMND_WLAN_IOCTL_GET_SCAN_RESULTS (0x0007) command, after the last scan has been stopped for at least 60 seconds. All further queries with HCI_CMND_WLAN_IOCTL_GET_SCAN_RESULTS (0x0007) will use Scan status 2 = No results until the scan is again started using HCI_CMND_WLAN_IOCTL_SET_SCANPARAM (0x0003) command. Scan status 2 = No results will also be output if the scan results are queried when not a single scan has been started after CC3000 power-up/reset.

RSSI (short for Received-Signal-Strength-Indication) field displays a unitless value in the range of 0 to 127 that characterizes the signal strength of an access point, with greater value meaning a stronger signal.

The WLAN security field has following options:

Value Binary Security type
0 b'00' No security/Open access
1 b'01' WEP
2 b'10' WPA
3 b'11' WPA2

The Entry time field displays the time at which this scan entry was internally recorded. It is an incremental second count, starting from the CC3000 power-up/reset.

HCI_EVNT_NVMEM_READ (0x0201)

HCI event operation code of HCI_EVNT_NVMEM_READ is the same as the calling command HCI_CMND_NVMEM_READ (0x0201).

The HCI_EVNT_READ_SP_VERSION message has 5 arguments:

  • HCI payload byte 1: status, always 0x00
  • HCI payload bytes 2:3: meaning unknown, values in this example 0x00 and 0x06 (as read out from CC3000)
  • HCI payload byte 4: package ID, value in this example 0x01 = 1

Note that the original TI host driver in SDK does not evaluate HCI payload bytes 2 (0x00) and 3 (0x06).

The HCI_EVNT_READ_BUFFER_SIZE message has 4 arguments:

  • HCI payload byte 1: status, always 0x00
  • HCI payload byte 2: the amount of buffers, value = 0x06 = 6
  • HCI payload bytes 3:4 (16-bit Little-endian value): buffer size = 0x05DC = 1500

Please note that the Buffer count field only says how many buffers CC3000 has, and not how many are still available (free). It is up to developer to keep track of used buffers in the code. At CC3000 power-up/reset all buffers reported by HCI_EVNT_READ_BUFFER_SIZE are available. Every successful call to HCI_DATA_SEND (0x81) or HCI_DATA_SENDTO (0x83) uses one buffer (if the call was successful is determined by parsing the message acknowledge HCI_EVNT_DATA_SEND (0x1003) or HCI_EVNT_DATA_SENDTO (0x100F) repectively). Buffers are released by CC3000 automatically, signalled using HCI_EVNT_DATA_UNSOL_FREE_BUFF (0x4100), having the amount of released buffers in the arguments.


HCI_EVNT_DATA_UNSOL_FREE_BUFF (0x4100)

Every time CC3000 is instructed to transmit data over the wireless link using HCI_DATA_SEND (0x81) or HCI_DATA_SENDTO (0x83), one of its internal buffers is consumed for temporarily saving the data to be transmitted. CC3000 has for this purpose a number of internal buffers (typically 6, but see HCI_CMND_READ_BUFFER_SIZE (0x400B) about how to read out the buffer count value on your specific CC3000). After CC3000 has successfully transmitted out the message, it will issue a HCI_EVNT_DATA_UNSOL_FREE_BUFF event to notify user code about buffers that can be used again for consequent transmissions.

It is up to user program to make sure that no further HCI_DATA_SEND (0x81) or HCI_DATA_SENDTO (0x83) messages are initiated if there are no buffers available anymore. For this, the user code should do bookkeeping about available buffer count by initializing the value at power-up/reset using the result of HCI_CMND_READ_BUFFER_SIZE (0x400B), then decrementing the value at every successfully initiated HCI_DATA_SEND (0x81) or HCI_DATA_SENDTO (0x83) message and increment the value again with amount of Buffers freed as replied by HCI_EVNT_DATA_UNSOL_FREE_BUFF.

Please also note that CC3000 might not immediately reply with HCI_EVNT_DATA_UNSOL_FREE_BUFF after it has transmitted the HCI_DATA_SEND (0x81) or HCI_DATA_SENDTO (0x83) message, but instead wait until there are only few available buffers or an internal timeout has been reached.
